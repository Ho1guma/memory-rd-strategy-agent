AI Agent Mini Project — 계획서

작성자: 최종민, 조성현, 유건욱, 손수민

1. 프로젝트 개요

  • 주제: HBM4, PIM(Processing-in-Memory), CXL 등 최신 반도체 R&D 정보 수집·분석
  • 분석 초점: 경쟁사별 기술 성숙도와 위협 수준 비교
  • 산출물: R&D 담당자가 즉시 활용 가능한 기술 전략 분석 보고서 (구조화된 문서)

2. A. 설계 — Agentic Workflow

2.1 Supervisor 역할

  • Supervisor 에이전트: 목표·성공 기준·계획 승인, 하위 에이전트 조율, 품질 게이트(재시도·분기), 최종 보고서 구조·톤 검수.
  • 실행 에이전트: 검색·추출·비교·초안 작성 등 Outcome 단위로 위임; Supervisor는 “무엇을 완료해야 다음 단계로 갈 수 있는가”만 판단.

2.2 Goal

  1. G1: 분석 대상 기술(HBM4/PIM/CXL)에 대해 의사결정에 쓰일 수 있는 최신 공개 정보가 체계적으로 수집·정리된 상태를 만든다.
  2. G2: 경쟁사별 TRL 추정·위협 수준을 동일한 기준으로 비교 가능한 형태로 정리한다.
  3. G3: 위 정보를 바탕으로 우선순위·대응 방향을 논의할 수 있는 기술 전략 분석 보고서를 완성한다.

2.3 Success Criteria

각 기준은 대상 / 변화 / 결정·행동 / 범위를 명시한다.
각 SC는 아래 품질 프레임워크 3축을 기준으로 설계한다.

  • 결과 품질 (Quality): 근거성(출처) · 완결성(누락 없음) · 일관성(내부 모순 없음) — TOOL로 자동 검증
  • 업무적 제약 (Constraints): Target Actor · Compliance · Scope Boundaries · Formatting
  • 서비스 기준 (Operational): Error Handling(Reflection) · Consistency · Stability — LOOP·Reflection 패턴으로 대응

G1 관련

  • SC1-1
    • 대상: HBM4, PIM, CXL 각각
    • 변화: 출처와 날짜가 붙은 요약 카드가 기술당 최소 5건 이상 확보되어야 하며, 근거 없는 스니펫은 포함하지 않는다.
    • 결정·행동: 기술 현황 섹션 작성 시 직접 인용 가능한 근거로 사용한다.
    • 범위: 논문, 학회, 특허, 공식 보도, IR 자료는 포함하고, 익명 커뮤니티의 단독 주장은 제외한다.
  • SC1-2
    • 대상: 수집 문서 메타데이터
    • 변화: keywords, entities 필드 존재 여부로 자동 판정하며, 추출 불가 문서는 "태깅 불가"로 표기한다.
    • 결정·행동: 검색, 재랭킹, 중복 제거에 사용한다.
    • 범위: 모든 수집 스니펫을 포함하되, 이미지 전용 PDF처럼 텍스트 추출이 불가능한 문서는 예외로 둔다.

G2 관련

  • SC2-1
    • 대상: 사전 정의된 경쟁사×기술 전체 조합
    • 변화: 모든 조합에 TRL 구간과 근거 2개 이상을 붙여야 하며, 근거가 부족한 경우에는 "정보 부족"을 명시한다.
    • 결정·행동: 경쟁사 비교 표와 서술의 근거로 사용한다.
    • 범위: TRL 4-6은 반드시 "추정" 라벨과 간접 지표를 함께 명시하고, 근거가 없으면 "정보 부족"으로 처리한다.
  • SC2-2
    • 대상: 동일 조합 전체
    • 변화: 위협 수준을 낮음, 중간, 높음으로 판정하되 TRL 값과 모순되지 않는 규칙을 유지한다.
    • 결정·행동: 전략 시사점의 우선순위를 논의할 때 사용한다.
    • 범위: 규칙은 README에 고정하며, 기본값은 TRL >= 7이면 높음, TRL 5-6이면 중간, TRL <= 4이면 낮음이다. 투자·채용 추세가 강하면 한 단계 상향할 수 있다.

일관성 판정: SC2-2의 위협 등급은 SC2-1의 TRL 수치와 교차 검증하여 모순 시 Supervisor가 Analysis 재실행 지시.

G3 관련

  • SC3-1
    • 대상: 최종 보고서
    • 변화: SUMMARY는 400자 이내, 본문은 4절 목차와 REFERENCE를 포함해야 하며, 모든 주장에 본문 [N] 인용이 있어야 한다. TRL 4-6 구간은 반드시 "추정" 한계를 고지하고, 전체 문체는 R&D 담당자 기준의 객관적이고 간결한 톤을 유지한다.
    • 결정·행동: R&D 리뷰 미팅 자료로 제출한다.
    • 범위: 한국어를 기본으로 하고 필요 시 영문을 병기하며, 회사명과 단위 표기를 통일하고 수율·원가에 대한 단정은 금지한다.
  • SC3-2
    • 대상: REFERENCE
    • 변화: 본문 [N] 인용과 URL, DOI, 제목, 접근일이 1:1로 대응되어야 하며, 매핑되지 않는 인용은 [매핑 불가]로 표시한다.
    • 결정·행동: 출처 검증이 가능하도록 유지한다.
    • 범위: 본문에서 실제로 인용한 자료만 포함하고, 단순 참고 자료는 제외한다.
  • SC3-3
    • 대상: 보고서 재실행 안정성
    • 변화: 동일한 evidence_store 입력이면 섹션 구조, TRL 구간, 위협 등급이 바뀌지 않아야 한다. 이는 temperature=0으로 보장한다.
    • 결정·행동: 품질 일관성을 유지한다.
    • 범위: 표현의 미세한 차이는 허용하지만, 섹션 구조나 등급 변경은 허용하지 않는다.

(N=5, 경쟁사 목록, 위협 등급 규칙은 scope.yaml 및 README에 수치·정의로 고정)

2.4 Plan — Task 분해

  1. T1: 웹, 논문, 특허, 문서에서 근거가 있는 스니펫을 수집해 메타데이터화한다. scope.yaml은 실행 전에 수동 설정한다.
    연결 SC: SC1-1, SC1-2 / 의존성: 없음
  2. T2: 로컬 또는 인덱스 기반 검색으로 질의별 관련 청크가 반환되는 상태를 만든다. 임베딩과 Retrieval 선정 평가(Hit Rate@K, MRR)는 이 단계 없이는 수행할 수 없으므로 사실상 필수다.
    연결 SC: SC1-2, Hit Rate@K/MRR / 의존성: T1
  3. T3: 기술별·회사별 TRL 추정표 초안을 만든다. 추정 구간은 명시한다.
    연결 SC: SC2-1 / 의존성: T1
  4. T4: 위협 수준 매트릭스 초안을 만든다.
    연결 SC: SC2-2 / 의존성: T3
  5. T5: 보고서 목차에 맞춘 통합 초안을 작성한다.
    연결 SC: SC3-1 / 의존성: T3, T4
  6. T6: REFERENCE 정합성을 맞추고 SUMMARY를 압축하며, 동일 입력 재실행 시 섹션 구조와 등급이 변하지 않는지 확인한다.
    연결 SC: SC3-1, SC3-2, SC3-3 / 의존성: T5

2.5 Control Strategy

  • 실행 전
    • 전략: 수동 설정
    • 설명: scope.yaml에서 technologies, competitors, keywords, n_evidence_min을 설정한다. 범위가 과도하면, 예를 들어 경쟁사가 5개를 넘거나 기술이 3개를 넘을 때 app.py가 자동 축소한다.
  • T1-T2
    • 전략: Loop + Retry (최대 3회)
    • 설명: SC1-1 미달이거나 중복률이 50%를 넘으면 쿼리를 재작성해 다시 검색한다. 3회를 초과하면 에스컬레이션한다.
  • Escalation
    • 전략: Abort + 인간 개입 요청
    • 설명: 최대 재시도 이후에도 SC를 충족하지 못하면, Supervisor가 수집된 증거와 함께 진행 불가 사유를 출력하고 중단한다.
  • TRL·위협
    • 전략: Human-in-the-loop (선택)
    • 설명: Supervisor가 SC2-1 기준상 근거 인용 수가 2개 미만인 경쟁사를 감지하면 T2로 되돌린다. 이 재실행은 재시도 카운터에 합산한다.
  • 최종
    • 전략: Linear
    • 설명: T6 이후 최종 단계를 순차 확정한다.

Supervisor SC 판정 메커니즘: Supervisor는 LLM 판단이 아닌 계측 가능한 수치를 우선 사용한다.

  • SC1-1: evidence_store의 출처별 레코드 수 카운트 ≥ N (자동 판정)
  • SC1-2: 메타데이터 내 keywords, entities 필드 존재 여부 (자동 판정)
  • SC2-1: TRL 표 내 각 경쟁사 행의 evidence_count ≥ 2 (자동 판정)
  • SC2-2: 위협 매트릭스 내 판정 규칙 필드 존재 여부 (자동 판정)
  • SC3-1/SC3-2: 목차 섹션 존재 + REFERENCE ↔ 본문 인용 매핑 검증 (자동 판정)
  • 수치 판정 통과 후 품질 검수(톤·논리) 단계에만 LLM 평가 사용

2.6 구조 (고수준)

Supervisor 패턴 핵심: 에이전트는 절대 다음 단계로 직접 이동하지 않는다. 반드시 Supervisor에게 결과를 반환하고, Supervisor가 SC 판정 후 다음 행동을 결정한다.

2.7 Retrieve 설계

Open-source Embedding 후보군 및 선정 기준

  • sentence-transformers 계열(multilingual 포함): 한·영 혼합 문서에 대응하기 좋다. 선정 시에는 한국어/영어 쿼리 재현율, 라이선스, 추론 비용을 본다.
  • BGE-m3 등 다국어 dense 모델: 긴 문맥과 다국어 환경에 강점이 있다. 같은 기준에 더해 벡터 차원과 메모리 사용량을 함께 본다.
  • Korean-specific 임베딩(선택): 국내 보도 비중이 높을 때 검토한다. 도메인 벤치, 즉 소규모 golden set에서의 성능을 기준으로 판단한다.

선정 절차: (1) golden 질의 세트 10–30개 작성 → (2) 후보별 임베딩·검색 파이프라인 동일 조건 비교 → (3) Hit Rate@K, MRR 기록 → 최종 모델·버전을 GitHub README에 반영.

Retrieval technique 후보군 및 선정 기준

  • Dense (vector): 임베딩 유사도를 사용하며, 의미적 매칭과 동의어 대응에 강하다.
  • BM25 / 키워드: 어휘 일치 기반이며, 제품명이나 표준명처럼 정확한 문자열 매칭이 중요할 때 유리하다.
  • Hybrid (dense + BM25, RRF 등): 두 방식을 결합해 종합 Hit Rate@K와 MRR을 높이는 방향으로 평가한다.
  • Re-ranking cross-encoder (선택): 상위 K개 결과를 재정렬해 정확도를 높일 수 있지만, 지연 비용과의 균형이 필요하다.

성능 기록: 개발 완료 후 README에 Hit Rate@K, MRR 및 평가 세트 요약(질의 수, K값) 명시.

2.8 Web Search (Exa) — 확증 편향 완화

  • 다각도 쿼리: 동일 주제에 대해 아래 쿼리 유형을 병렬 실행한다.

    • 현황: "{기술명} latest development {연도}"
    • 한계·비판: "{기술명} limitations challenges"
    • 경쟁·대안: "{기술명} vs {대안기술} comparison"
    • 채용·투자 신호: "{기업명} {기술명} hiring investment"
    • 반증: "{기술명} failed abandoned setback"

  • 출처 다양성: 단일 도메인 출처가 전체 인용의 40% 초과 시 Supervisor가 T2 재실행 트리거.

  • Exa: API 기반 웹 검색 도구로 구현; 검색 결과는 {url, title, date, snippet, domain} 메타데이터와 함께 evidence_store에 저장.

2.9 논문·특허 검색 (OpenAlex + Lens) — 근거 강화

  • 논문 검색: OpenAlex를 1차 API로 사용해 논문·학회·저자·기관 메타데이터를 수집한다. 오픈 메타데이터의 일관성과 대량 처리 안정성을 우선한다.
  • 특허 검색: Lens를 1차 API로 사용해 특허 패밀리, 인용, 출원인, 패밀리 단위 변화를 추적한다. 특허 커버리지와 분석 편의성이 강점이다.
  • 폴백 전략: OpenAlex가 비어 있거나 커버리지가 약한 경우 Semantic Scholar / Crossref를, Lens 접근성이 제한되거나 일부 관할 특허가 누락되는 경우 PatentsView / Google Patents 계열을 보조로 사용한다.
  • 판정 기준: 논문·특허 결과는 웹 검색과 별도 evidence lane으로 저장하고, source_type, doi, patent_number, assignee, publication_date 같은 필드를 표준화한다.

3. B. 기술 전략 분석 보고서

3.1 보고서 산출물 구성안

최종 보고서는 아래 목차를 기본 골격으로 구성한다. 이 섹션은 실제 산출물의 챕터 구조를 설계서 차원에서 미리 고정하기 위한 것이다.

  1. SUMMARY
    • 보고서 전체 핵심 내용을 1/2 페이지 이내로 요약한다.
  2. 0. 앞페이지
    • sk hynix 보고서 포맷을 따른 표지 또는 앞페이지를 둔다.
  3. 1. 분석 배경
    • 왜 지금 이 기술을 분석해야 하는지를 설명한다.
    • 세부 구성은 다음을 포함한다.
    • 시장 동향 및 배경: 텍스트 문단 형식으로 정리한다.
    • 그림/도표: 수요량 등 핵심 지표를 시각 자료로 제시한다.
  4. 2. 분석 대상 기술 현황
    • 현재 기술 수준과 개발 방향을 sk hynix 중심으로 정리한다.
    • 세부 구성은 1) HBM, 2) PIM, 3) CXL로 나눈다.
    • 각 도메인에는 자사 기준 영업이익, 점유율, 연구 수준을 함께 포함한다.
  5. 3. 경쟁사 동향 분석
    • 경쟁사의 기술 전략과 최신 움직임을 기술별로 나누어 정리한다.
    • 기본 구조는 1) HBM, 2) PIM, 3) CXL 순서로 통일한다.
    • 각 기술 섹션은 경쟁사 비교표와 sk hynix를 함께 제시하는 요약 파트를 먼저 둔다.
    • 이후 필요하면 회사별 세부 내용을 선택적으로 붙인다.
    • 예시 구조는 다음과 같다.
    • 1-1) 경쟁사 비교표 + sk hynix
    • 1-2) 회사1 세부내용 (optional)
    • 1-3) 회사2 세부내용 (optional)
    • 1-4) 회사3 세부내용 (optional)
    • 1-5) 비주류 회사 세부내용 (optional)
  6. 4. 장기적인 관점에서의 새로운 개념 또는 신기술 (optional)
    • 신기술, 차세대 메모리 개발 동향 등 장기 관점의 탐색 주제를 다룬다.
  7. 5. 전략적 시사점
    • R&D 우선순위 관점에서 대응 방향을 제언한다.
  8. REFERENCE
    • 보고서 작성 시 활용한 자료 목록을 정리한다.

3.2 보고서 목차 설계 원칙

  • 목차는 배경 → 자사 기술 현황 → 경쟁사 비교 → 장기 신기술 → 전략적 시사점 흐름을 유지한다.
  • sk hynix 관점의 내부 활용 문서라는 점을 반영해, 자사 현황과 경쟁사 비교가 분리되되 자연스럽게 이어지도록 설계한다.
  • 기술별 구성(HBM, PIM, CXL)은 가능한 한 같은 순서와 같은 하위 구조를 유지해 독자가 기술 간 비교를 쉽게 할 수 있게 한다.
  • 경쟁사 섹션은 요약 비교표를 먼저 두고, 상세 서술은 선택적으로 확장하는 방식으로 설계해 보고서 길이와 밀도를 조절할 수 있게 한다.
  • 장기 신기술 섹션은 필수 본문과 분리해 optional로 두어, 프로젝트 범위와 수집 근거 수준에 따라 유연하게 포함 여부를 결정한다.

3.3 전략 비교 — TRL 기반

  • TRL 1–9 정의는 가이드와 동일하게 보고서 부속 또는 본문에 요약 인용.
  • 공개 정보 기반 추정
    • TRL 1–3: 논문·학회·특허 중심 근거.
    • TRL 4–6: 정보 갭 큼 → 수치 단정 금지, “추정” 및 간접 지표(특허 출원 패턴, 학회 발표 빈도 변화, 채용 키워드 등) 명시.
    • TRL 7–9: 양산·실적·샘플 공급 등 공개 자료 위주, 단 수율·원가는 비공개 전제 명시.
  • 한계 고지: 에이전트 성능과 무관하게 TRL 4–6은 추정 영역임을 보고서 본문에 명시적으로 기술.

3.4 SWOT vs TRL

  • SWOT는 상대·주관적 비교에 유리.
  • TRL은 “현재 단계” 절대 위치 제공; 보고서에서는 TRL을 주축으로 표를 만들고, 필요 시 SWOT는 보조 섹션으로 제한.

4. State 필드 정리

이 절에서는 구현 시 공통으로 관리할 핵심 State만 정리한다. 워크플로 요약이나 에이전트 역할은 앞 절에서 이미 설명했으므로 여기서는 반복하지 않는다.

4.1 State 필드 정의

  • scope (object): scope.yaml 로드값. technologies, competitors, keywords, n_evidence_min 등을 포함한다.
  • evidence_store (list[dict]): 수집 스니펫 저장소. url, title, date, snippet, domain, keywords, entities 등을 담는다.
  • iteration_count (int): 현재 T1 재시도 횟수이며 최대 3회까지 허용한다.
  • sc_status (dict): SC별 자동 판정 결과. SC1-1부터 SC3-2까지 pass, fail, count 형태로 관리한다.
  • trl_table (list[dict]): 경쟁사별 TRL 추정 결과. company, technology, trl_range, evidence_count, label 등을 포함한다.
  • threat_matrix (list[dict]): 경쟁사별 위협 등급 결과. company, level, rationale을 포함한다.
  • draft_report (str): 보고서 Markdown 초안이다.
  • reference_list (list[dict]): 본문 인용과 출처 매핑 목록이다. citation_id, url, title, accessed_date 등을 포함한다.
  • last_error (str): 최근 오류 메시지이며, 없으면 null이다.

5. 프롬프트 품질 기준

LLM 호출 시 아래 3가지 기준을 Analysis·Report 에이전트 프롬프트에 명시적으로 반영한다.

5.1 결과 품질 기준 (Quality)

  • 근거성: 모든 주장에는 출처가 있어야 한다. 구현 측면에서는 증거 자료에 없는 내용을 창작하지 못하도록 명시하고, evidence_refs에는 실제 URL만 허용하며, 근거가 없으면 반드시 "정보 부족"으로 표기한다.
  • 완결성: 필수 항목이 빠짐없이 포함되어야 한다. 경쟁사×기술 전체 조합을 모두 커버하도록 요구하고, 위협 매트릭스에도 모든 경쟁사 행이 생성되도록 강제한다.
  • 일관성: 내부 모순이 없어야 한다. 이를 위해 위협 등급이 TRL 수치와 충돌하지 않도록 프롬프트에 교차 검증 규칙을 명시한다.

5.2 업무적 제약 조건 (Constraints)

  • Target Actor: 독자는 반도체 R&D 담당자와 기술 전략 임원 같은 고급 전문가다. 따라서 문체는 객관적이고 간결하며 논거 중심이어야 하고, SUMMARY는 400자 이내로 제한한다.
  • Compliance: TRL 4-6은 반드시 "추정"이라고 명시해야 하며, 수율과 원가를 단정해서는 안 된다. 증거 없는 수치 생성도 금지하고, 매핑되지 않는 인용은 [매핑 불가]로 분명히 표시한다.
  • Scope Boundaries: 포함과 제외 범위는 scope.yaml 기준으로 고정한다.
  • Formatting: TRL과 위협 출력은 JSON 배열로 다루고, 보고서는 Markdown으로 작성하며, 회사명과 기술 용어 표기를 통일한다.

5.3 서비스 기준 (Operational)

  • Error Handling: JSON 파싱이나 필드 검증이 실패하면 피드백을 포함해 최대 2회까지 재요청한다. 구현은 _llm_call_with_retry를 사용하고, LLM API 오류는 try/except 처리 후 last_error에 전달한다.
  • Consistency: temperature=0을 고정하고, _validate_trl_row, _validate_threat_row 같은 필수 필드 검증을 수행한다.
  • Stability: evidence_store가 비어 있어도 빈 결과와 경고를 반환해 그래프 전체가 중단되지 않도록 한다.

6. 리스크·완화

  • TRL 4-6 과신 서술: 추정 라벨, 간접 지표, 한계 절을 필수로 두고 Compliance 규칙에도 같은 내용을 명시한다.
  • 확증 편향: Exa 다각도 쿼리, 반증 쿼리, 출처 다양성 규칙으로 완화한다.
  • 환각 인용: REFERENCE와 본문을 교차 검증하고, 매핑 불가 인용은 자동 감지해 표기하며, 원문 스니펫을 함께 보관한다.
  • 범위 폭발: T1 단계에서 경쟁사 수, 기술 수 상한, SC1-1 최소 건수만 조정한다.
  • JSON 파싱 실패: _llm_call_with_retry 재시도와 필드 검증으로 빈 결과 생성을 방지한다.
  • LLM API 오류: try/except로 처리한 뒤 last_error를 기록하고, 이후 Supervisor가 에스컬레이션한다.
